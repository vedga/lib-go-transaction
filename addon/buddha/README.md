Пакет buddha предназначен для перемещения активных (не выполненных до конца) задач между нодами, на которых они могут выполняться.

Для этого создаем один экземпляр Колеса Сансары вида
```go
samsara := New(func(ctx context.Context, tx transaction.Transaction) error {
	// Prepare migration data
	migrationData, e := tx.Encode()
	if e != nil {
		return e
    }
	
	return SendDataViaMessageBus(ctx, migrationData)
})
```

Сами задачи создаем так же, как и задачи для транзакций (т.е. определяем структуру контекста задачи и определяем для нее метод Run), но в менеджере транзакций используем опцию определения создателя задач WithTxTaskProducer() из пакета buddha, которое также на входе принимает Колесо Сансары в качестве аргумента:
```go
manager := transaction.NewManager(
	// Task A
	// Initialization difference:
	// instead of transaction.WithTxTaskProducer(...) use buddha.WithTxTaskProducer(...)
	buddga.WithTxTaskProducer(kindA, samsara, func(setup ...data.Setup) (transaction.Task, error) {
		producer := data.NewProducer[taskA]()

		task, e := producer(append([]data.Setup{
			// Имплементация конструктора task A
			data.NewSetup[taskA](func(o *taskA) error {
				// Инициализация контекста task A
				return nil
			}),
		}, setup...)...)
		if e != nil {
			return nil, e
		}

		return data.As[transaction.Task](task)
	    }),
    )
```

Колесо Сансары имплементирует интерфейс io.Closer(), который надо вызывать для переселения всех активных задач на другую исполняемую ноду. При вызове метода Close() Колеса Сансары для всех активных задач произойдет отмена контекста в методе Run(). Если операции задачи еще не завершены, необходимо вернуть код ошибки transaction.ErrMigrate, по которому в транзакцию будет сохранен текущий контекст задачи и транзакция будет передана в функцию, заданную в конструкторе Колеса Сансары для передачи ее на шину сообщений. 

После приема сообщения на другой ноде контекст задачи будет восстановлен и для нее будет вызван метод Run(). Следует отметить, что какая-то часть работы уже может быть выполнена, для этого контекст задачи сохраняется перед передачей. Поэтому в методе Run() задачи необходимо по контексту (к примеру при помощи finite state machine) определить, какая часть работы уже была выполнена, и продолжить работу именно с этой точки.

Переселение задач возможно не ограниченное кол-во раз до полного выполнения всего потока психофизических элементов (скандх). Индикатором выполнения жизненного цикла является возврат кода ошибки nil из метода Run() задачи, после чего она наконец-то попадает в Нирвану.
