Предлагаемый фреймворк предоставляет возможность распределенного выполнения задач в транзакционном режиме. Также поддерживается outbox-паттерн для гарантированной отправки сообщения при фиксировании транзакции.

Основная идея заключается в следующем: все сервисы получают сообщения из единой шины данных. Если сервис умеет обрабатывать текущую задачу в очереди, то он запускает ее на выполнение. Если сервис не умеет обрабатывать текущую задачу, то полученное сообщение им игнорируется (ожидается, что задача будет обработана тем сервисом, который умеет это делать). Каждая задача во время выполнения также может формировать список задач для отката транзакции путем помещения их на стек отката через метод QueueRollbackTask() транзакции. Кроме того задача может положить произвольные данные на стек данных, которые могут быть в дальнейшем обработаны последующими задачами транзакции.

Во время выполнения задачи в ней может возникнуть восстановимая или не восстановимая ошибка. В случае восстановимой ошибки задача должна вернуть ошибку, созданную при помощи NewRetryTaskError(maxRetryCount). В этом случае исходная транзакция будет повторно помещена на шину данных, но в ней увеличится счетчик попыток выполнения текущей задачи. Счетчик попыток доступен для чтения через свойство Attempt() транзакции (в данный момент не работает!). В случае возникновения невосстановимой ошибки задача должна вызвать метод Rollback(), который инициирует откат транзакции.

Для асинхронного выполнения задач можно по входным параметрам формировать контекст задачи и добавлять его в транзакцию методом QueueTask() или QueueEncodedTask().
